Overview

Why We Need a

Recovery Plan

Our AIX build servers are critical infrastructure. When they go down, builds stop and work halts. This plan ensures we are always prepared to recover — fast and consistently — no matter what type of disaster occurs.

Disaster Scenarios We Plan For
⚙️

Hardware Failure

Physical server components — disks, adapters, processors, or memory — may fail, preventing normal operation. Recovery may require hardware repair or restoration to an alternate LPAR before system recovery can begin.

💾

OS Corruption

System files, rootvg corruption, failed updates, or software issues may render the operating system unusable. The affected system can be restored using an existing mksysb image.

🗑️

Accidental Deletion

Critical files, directories, or application data may become corrupted or deleted. Depending on scope, either partial recovery or full system restoration may be required.

🔌

Storage or Site Outage

Power failures, network outages, or environmental incidents may affect multiple systems simultaneously. Recovery must account for both system restoration and supporting services.

Phase 1

Backup Process

Two Types of Backups
mksysb — OS Backup

Root Volume Group (rootvg)

A full snapshot of the operating system — everything the machine needs to boot. rootvg is mostly static and rarely changes, so one monthly backup is sufficient to recover the entire OS to its last known good state.

Once a Month
via SMIT on client
savevg — Data Backup

Non-Root Volume Groups

Backs up application data, build artifacts, and user files. This data changes constantly with every build cycle, so weekly backups are required to minimize the amount of work that could be lost in a disaster.

Every Friday Night
via SMIT on client
How Backups Flow to the NIM Master
1

Generate the backup locally on the client

Use Grover's script to create the correct backup and to detect the state of the system.

2

Use Grover's Script to write to distbkup2 (NIM Master)

Once the backup is created, the client uses Grover's script to write the image file directly into the designated directory on the NIM Master. This is the central backup repository.

3

NIM Master creates 3 recovery resources

From the uploaded image, NIM Master creates: ① the mksysb resource, ② the SPOT resource (network boot environment), and ③ the NIM Client Definition (MAC address, CPU ID, machine ID).

Retention Policy — 1 Month Rolling

Keep the 4 most recent weekly backups per client (1 full month). On Week 5, delete the oldest set before adding the new one. Always monitor NIM Master disk space.

04B / 07
Backup Tooling

What is Grover's Script?

What It Does
✍️
Writes Directly to distbkup2
Instead of saving the mksysb image or savevg backup to the client's local machine first, Grover's script writes the backup directly into the NIM Master (distbkup2). This eliminates the need for local storage on the client and removes the manual SCP transfer step entirely.
📅
Scheduling Built In
Grover's script supports automated scheduling, allowing backups to be set to run at the desired time (e.g. Friday nights) without manual intervention — keeping the backup process consistent and hands-off.
Build Protection Logic
Smart Freeze & Resume
1
A backup is scheduled and starts to run on the client
2
If a build is actively running, the script freezes the backup and takes a snapshot of the current system state
3
The build is allowed to finish running uninterrupted — no disruption to active work
4
Once the build completes, the script resumes and continues the backup from where it left off
Key Benefit
Grover's script ensures that backups never interrupt active builds, and builds never block backups — the two processes coexist safely and automatically.
05 / 07
Phase 2

Recovery Process

01

Resolve Hardware First

If the disaster involves a hardware failure, the physical issue must be resolved before recovery can begin. If the original hardware cannot be repaired immediately, recovery can be performed on an alternate LPAR with similar disk capacity and resources.

→
02

Allocate NIM Resources

On the NIM Master, use smitty nim to allocate the three required resources — the mksysb image, the SPOT resource, and the client definition — to the target client. Confirm allocation with: lsnim -l <clientname>

→
03

Boot Client into SMS Menu

Reboot the client system and stop it at the SMS (System Management Services) menu — the low-level boot interface. From SMS, select the Ethernet adapter as the boot device. If the adapter name is unknown, identify it by MAC address.

→
04

Network Boot & Install

The client boots over the network and connects to the NIM Master. It pulls down the allocated mksysb and SPOT resources. The system boots into the AIX installation menu where the operator selects the target disk. Installation proceeds automatically.

→
05

Verify & Confirm Recovery

After installation completes: verify the system boots correctly, confirm all applications and services function as expected, validate data integrity, and ensure the NIM client definition matches the restored system.

✓ Recovery Complete
06 / 07
NIM Deep Dive

Three Resources

Every Recovery Needs

Before a recovery can begin, the NIM Master must have all three resources created and correctly allocated to the target client. Without all three in place, the network installation will not proceed. These resources are created from the backup image transferred by the client.

1

mksysb Resource

Points directly to the mksysb backup file that was written to the NIM Master via Grover's script. This is the core OS image used during restoration — it contains everything needed to reinstall the operating system to its exact pre-disaster state, including all system configuration.

Created from: the client's mksysb file written to NIM Master via Grover's script
2

SPOT Resource (Shared Product Object Tree)

Extracted directly from the mksysb resource. The SPOT provides the /usr filesystem environment that the client needs in order to perform a network boot. Without a SPOT resource, the client has no environment to boot into and the NIM installation cannot start.

Created from: the mksysb resource — Step 1 must be completed first
3

NIM Client Definition

Identifies the target system to the NIM Master. Includes the MAC address, CPU ID, and machine ID (obtained by running uname -m on the client). The more accurate and complete this definition, the smoother the communication between NIM Master and the client during recovery.

Verify all resources allocated correctly: lsnim -l <clientname>
